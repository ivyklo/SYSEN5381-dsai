# Database Schema – City Congestion Tracker

This document describes the **PostgreSQL schema** and **data structure** used by the City Congestion Tracker.

---

## 1. Tables

### 1.1 `locations`

One row per monitored location (intersection, segment, or zone).

```sql
CREATE TABLE IF NOT EXISTS locations (
  location_id   TEXT PRIMARY KEY,
  name          TEXT NOT NULL,
  zone          TEXT,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
```

- **`location_id`** (`TEXT`, PK): Stable identifier used as a foreign key from `congestion_readings`  
  Examples: `downtown_1`, `highway_e`, `bridge_n`.
- **`name`** (`TEXT`): Human-readable label for UI and reports  
  Examples: `"Main & 5th"`, `"Highway 9 East"`.
- **`zone`** (`TEXT`): Logical grouping / area  
  Examples: `downtown`, `highway`, `bridge`, `transit`.
- **`created_at`** (`TIMESTAMPTZ`): Row creation timestamp (set by the database).

Populated by `scripts/generate_synthetic_data.py`.

---

### 1.2 `congestion_readings`

One row per congestion reading (location + timestamp).

```sql
CREATE TABLE IF NOT EXISTS congestion_readings (
  id              BIGSERIAL PRIMARY KEY,
  location_id     TEXT NOT NULL REFERENCES locations(location_id) ON DELETE CASCADE,
  ts              TIMESTAMPTZ NOT NULL,
  congestion_level INTEGER NOT NULL CHECK (congestion_level BETWEEN 1 AND 4),
  delay_minutes   NUMERIC(5,2),
  dataset_label   TEXT DEFAULT 'default',
  created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

- **`id`** (`BIGSERIAL`, PK): Surrogate key.
- **`location_id`** (`TEXT`, FK): References `locations.location_id`.  
  When a location is deleted, its readings are removed via `ON DELETE CASCADE`.
- **`ts`** (`TIMESTAMPTZ`): Timestamp when the reading was taken (15-minute interval).
- **`congestion_level`** (`INTEGER`): Discrete congestion category:
  - `1` = low  
  - `2` = moderate  
  - `3` = high  
  - `4` = severe  
  Enforced by `CHECK (congestion_level BETWEEN 1 AND 4)`.
- **`delay_minutes`** (`NUMERIC(5,2)`): Estimated delay for this reading in minutes (derived from `congestion_level` with noise).
- **`dataset_label`** (`TEXT`, default `'default'`): Identifies the **test dataset** a row belongs to:
  - `dataset_a` – last **30 days** of data (Test A)  
  - `dataset_b` – last **14 days** of data (Test B)  
  - `dataset_c` – last **7 days** of data (Test C)  
  Used by the dashboard’s “Refresh Test A/B/C data” buttons and FastAPI filters.
- **`created_at`** (`TIMESTAMPTZ`): Row creation timestamp (set by the database).

Indexes defined in `supabase/schema.sql`:

```sql
CREATE INDEX IF NOT EXISTS idx_readings_location ON congestion_readings(location_id);
CREATE INDEX IF NOT EXISTS idx_readings_ts ON congestion_readings(ts);
CREATE INDEX IF NOT EXISTS idx_readings_level ON congestion_readings(congestion_level);
CREATE INDEX IF NOT EXISTS idx_readings_dataset ON congestion_readings(dataset_label);
```

These support fast filtering by location, time window, congestion level, and dataset.

## 2. Data Generation & Layout

Synthetic data is generated by **`scripts/generate_synthetic_data.py`**:

- Uses 7 base locations (stored in `locations`).
- Generates **three test datasets**:
  - **Test A (`dataset_a`)** – last 30 days, 15-minute intervals
  - **Test B (`dataset_b`)** – last 14 days, 15-minute intervals
  - **Test C (`dataset_c`)** – last 7 days, 15-minute intervals
- Writes CSVs into `data/`:
  - `data/locations.csv`
  - `data/congestion_readings.csv` (A/B/C combined, includes `dataset_label`)
- If `SUPABASE_DB_*` is set in `.env`, it also:
  - Truncates `congestion_readings`
  - Refreshes `locations`
  - Inserts all synthetic readings with the correct `dataset_label`.

## 3. How the API Uses the Schema

The FastAPI backend (`api/fastapi_app.py`) queries this schema:

- **`GET /locations`** – `SELECT location_id, name, zone FROM locations ORDER BY location_id`
- **`GET /readings`** – joins `congestion_readings` to `locations` for `location_name`, with filters:
  - `dataset` → `dataset_label = :dataset`
  - `from_ts`, `to_ts` → `ts BETWEEN :from_ts AND :to_ts`
  - `location_id` list → `location_id = ANY(:location_ids)`
  - `min_level`, `level` list → (optional congestion filters)
- **`GET /summary`** – groups by `location_id` to compute `n`, `avg_level`, `max_level`, optionally filtered by `dataset`, time range, and levels.
- **`POST /ai-summary`** – uses the same grouping as `/summary` and passes condensed stats into OpenAI.

The dashboard never queries the database directly; it always goes through the API, which relies on the schema above.
